# Decisions

本目录用于沉淀 `openclaw-guardian` 已经达成共识的正式决策。

这里记录的不是候选方案，也不是尚未定稿的 proposal，而是：

- 已接受的方向
- 已确认的边界
- 后续实现应遵守的规则

## 1. 什么内容适合进入本目录

适合进入 `docs/decisions/` 的内容包括：

- 能力层边界已经明确的架构决策
- issue 治理方式已经确定的产品决策
- 语言规范、工作流规范等已经稳定的治理决策
- 影响多个 issue 或公共层的长期规则

## 2. 什么内容不应进入本目录

下面这些内容不应直接进入本目录：

- 尚未公开讨论的草稿
- 仍在比较中的多个方案
- 只停留在“想法”层面的 proposal
- 纯执行任务
- 仍需要大量探索的实现细节

这类内容应先通过：

- 维护工作区草稿
- `GitHub Discussions`

再决定是否进入 `docs/decisions/`。

## 3. 推荐命名方式

建议使用：

```text
YYYY-MM-DD-<topic>.md
```

例如：

- `2026-03-25-direct-session-reset-policy.md`
- `2026-03-25-introduce-improve-capability.md`
- `2026-03-25-session-handoff.md`

## 4. 推荐结构

每篇 decision 建议至少包含下面这些部分：

1. 背景
2. 决策
3. 适用范围
4. 不采纳的备选方案
5. 对用户和维护者的影响
6. 后续实现要求

如果该决策来自某条 `Discussion`，建议额外注明：

- 对应的 `Discussion` 链接
- 达成共识的日期

## 5. 与 `Issues` 的关系

`docs/decisions/` 不是任务跟踪系统。

关系应当是：

- `Discussion`
  负责公开讨论方案
- `docs/decisions/`
  负责沉淀已接受的决策
- `Issue`
  负责跟踪具体实现任务

## 6. 语言规范

本目录默认使用简体中文。

下列内容保持英文，并使用反引号包裹：

- 术语
- 路径
- 能力名
- issue id
- 字段名
- 命令

## 7. 当前阶段的约定

在某项 proposal 还没有进入本目录之前：

- 不要把它当成既定事实写入实现说明
- 不要据此直接扩展公共层
- 不要把草案长期保留在 repo 的其他目录中冒充正式文档

换句话说：

- proposal 先讨论
- 共识再决策
- 决策之后才实施
