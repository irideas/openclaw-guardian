# Discussion To Decision Workflow

本文档定义 `openclaw-guardian` 在 proposal、公开讨论、共识沉淀与实现跟踪之间的标准流转。

目标是让维护者与 AI Agent 在面对新问题、新能力层或新治理规则时，知道：

- 什么内容应先走 `GitHub Discussions`
- 什么内容可以直接开 `Issue`
- 什么时候应沉淀到 `docs/decisions/`
- 这些阶段分别使用什么语言、结构和输出形式

## 1. 适用范围

下面这些内容，默认先走 `Discussions`：

- 新的 issue 候选问题
- 新的能力层命名与边界讨论
- 运行时策略调整建议
- 平台增强方向
- 维护工作流、治理规则与公开规范
- 需要比较多个方案才能收敛的设计题

下面这些内容，可以直接开 `Issue`：

- 已有共识、范围明确、可以直接实施的改动
- 明确的 bug 修复
- 已经接受的决策的实现任务
- 明确的文档补齐、测试补齐或工程清理任务

## 2. 标准流转

项目默认采用下面的流转：

1. 工作区草稿
2. `GitHub Discussions`
3. 讨论达成共识
4. 沉淀到 `docs/decisions/`
5. 如需实现，再创建 `Issue`
6. 通过分支、提交、PR 和测试完成落地

这意味着：

- 未定稿的 proposal 默认不进入 repo 已跟踪内容
- repo 中不长期保存“尚未达成共识”的 proposal 文档
- repo 中的 `docs/decisions/` 只承载已经接受的决策

## 3. 各阶段的推荐载体

### 3.1 工作区草稿

工作区草稿用于：

- 收集证据
- 整理反例
- 起草 `Discussions` 正文
- 比较多个候选方案

工作区草稿的特点：

- 可以快速迭代
- 不进入 repo 已跟踪内容
- 不视为公开承诺

### 3.2 `GitHub Discussions`

`Discussions` 是当前阶段的正式公开提案载体。

适合发布：

- 候选问题
- 增强方向
- 能力层设计
- 工作流和治理提案

不适合发布：

- 已经明确排期的纯执行任务
- 只差动手实现的小修复

### 3.3 `docs/decisions/`

当讨论已经达成共识时，应把结论沉淀到 `docs/decisions/`。

这里记录的是：

- 已采用的方案
- 不再继续争论的边界
- 后续实现应遵守的决策依据

### 3.4 `Issues`

只有在方案已经清晰、可直接实施时，才应创建 `Issue`。

`Issue` 的重点是：

- 任务范围
- 实现目标
- 验收标准
- 跟踪与关闭

## 4. 语言规范

### 4.1 `Discussions`

`Discussions` 正文默认使用简体中文。

下列内容保持英文，并使用反引号包裹：

- 术语
- 文件路径
- 代码标识符
- issue id
- 能力名
- 配置字段
- 命令

例如：

- `preflight`
- `repair`
- `mitigation`
- `improve`
- `docs/decisions/`
- `session.resetByType.direct`

### 4.2 `docs/decisions/`

决策文档默认也使用简体中文，术语保持英文。

### 4.3 `Issues`

如果 `Issue` 来源于已经接受的中文讨论，则 `Issue` 也应优先使用简体中文描述，必要时附带精简英文术语，而不是反过来。

## 5. `Discussions` 最小结构

每篇 `Discussion` 至少应包含：

1. 背景
2. 当前问题或机会点
3. 现象与证据
4. 根因或动机分析
5. 候选方案
6. 推荐方案
7. 开放问题
8. 建议下一步

如果是能力层提案，建议额外包含：

- 与现有能力层的边界
- 为什么不能并入现有能力
- 默认开关与用户心智

## 6. 何时视为达成共识

满足下面条件时，可视为已经具备“沉淀到 `docs/decisions/`”的基础：

- 主要边界已经清楚
- 关键命名已稳定
- 已经排除明显更优的替代方案
- 维护者之间对目标和方向没有主要分歧
- 后续实现已经可以按该方向推进

注意：

- “有人提出了一个可行方案”不等于“达成共识”
- “已经开始实现”也不自动等于“达成共识”

## 7. 何时进入代码实现

只有在出现下列条件时，才建议从讨论转入实现：

- 已有 `docs/decisions/` 决策文档
- 实现目标可被拆成明确任务
- 影响范围基本清楚
- 测试与文档补齐路径可预期

如果还处在下列状态，应继续留在讨论阶段：

- 命名仍在摇摆
- 方案边界仍然模糊
- 用户心智还没有统一
- 具体实现方式仍需要大量探索

## 8. 对 `internal/` 的当前约定

当前工作流下：

- 不再把新增 proposal、候选方案或治理草稿优先放入 repo 的 `internal/`
- 未定稿草稿应先保留在维护工作区
- 公开提案以 `GitHub Discussions` 为准
- 已接受结论进入 `docs/decisions/`

如果未来确有必要引入新的 repo 内部目录，应先通过 `Discussions` 明确其边界和用途。

## 9. 推荐实践

- 先把“问题是什么”说清，再讨论“代码怎么改”
- 先把“为什么不能并入现有能力”说清，再引入新能力层
- 先形成共识，再进入实现跟踪
- 对用户可见的提案，优先使用简体中文，避免维护讨论变成只适合少数人阅读的英文碎片
