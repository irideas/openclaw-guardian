# Discussions 工作台使用规则

本文档定义 `openclaw-guardian` 当前 GitHub Discussions 工作台的分工边界，避免不同类型的信息混写。

## 1. 目标

Discussions 不是随手聊天区，而是项目维护过程中的结构化工作台。

它主要承载：

- 阶段性维护记录
- 候选问题归并
- Owner 决策留痕
- 周期性维护回顾

不适合承载：

- 应进入 `docs/` 的对外公开说明
- 应进入 `internal/` 的长期治理规则正文
- 已经足够明确、应直接落到 issue 的正式问题描述

## 2. 当前四个核心 Discussions 的职责

### 2.1 guardian-chief 工作台

入口：`Discussions #1`

用途：

- 记录 `guardian-chief` 的阶段性维护动作
- 同步工作台调整、流程收口、维护方式变化
- 记录值得留痕、但暂不适合进入 issue / owner decision / weekly review 的项目维护事项

适合写入：

- 本轮做了哪些治理整理
- 为什么新增某个内部规则或工作台约定
- 某次维护收口的简要说明
- 工作方式、载体、协作习惯的阶段性更新

不适合写入：

- 单个候选问题的详细证据归并
- 需要 Owner 拍板的正式决策结论
- 每周回顾正文

### 2.2 候选问题管道 / candidate pipeline

入口：`Discussions #2`

用途：

- 归并新出现的问题线索
- 维护“是否值得纳入 guardian”的前置判断
- 为后续 issue 化做证据收束

适合写入：

- 新问题线索摘要
- 来源、复现条件、影响范围、已知版本范围
- 与既有候选项的合并判断
- 初步结论：继续观察 / 形成候选 / 放弃 / 等待上游

不适合写入：

- 已经明确成立、应直接进入正式 issue 的最终内容
- 纯项目治理规则
- 与候选问题无关的维护播报

### 2.3 Owner 决策记录 / owner decisions

入口：`Discussions #3`

用途：

- 留痕必须由 Owner 拍板的事项
- 把分析建议与最终决策分开记录

适合写入：

- 是否正式纳入某个新 guardian issue
- 是否采用重要 `mitigation` / `repair` 策略
- 是否发布版本
- 是否调整项目边界、身份策略、部署形态

建议写法：

- 先写背景与备选项
- 再写建议判断
- 最后明确 Owner 决策结果

不适合写入：

- 仅需 agent 自行执行的普通维护动作
- 没有决策点的日常播报

### 2.4 每周维护回顾 / weekly maintenance review

入口：`Discussions #4`

用途：

- 沉淀固定节奏的周回顾
- 汇总一周内的线索、动作、判断与下一步建议

适合写入：

- 本周新增问题线索
- 候选问题进展
- 上游变化与社区案例
- 文档、测试、版本范围、治理规则的调整
- 下周建议关注项

不适合写入：

- 零散即时更新
- 单条候选问题的细节归并
- 需要立即拍板的决策单

## 3. 使用顺序建议

遇到新信息时，优先按下面顺序判断落点：

1. 如果是对外长期说明，放 `docs/`
2. 如果是内部长期规则，放 `internal/`
3. 如果是正式问题现象，进入 `issues/` 或其草稿过程
4. 如果还是阶段性维护过程信息，再判断是否进入 Discussions

进入 Discussions 后，再继续分流：

- 工作方式/维护动作同步 → `guardian-chief 工作台`
- 问题线索归并 → `candidate pipeline`
- 需要 Owner 决策 → `owner decisions`
- 周期性回顾总结 → `weekly maintenance review`

## 4. 记录风格

当前统一约定：

- 默认使用简体中文
- 先写判断，再写事实和依据
- 不把暂不确定的现象写成确定结论
- 一条记录尽量只服务一个主要目的，避免混写
- 能沉淀到 repo 正文的正式事实，最终仍应回到仓库文件中

## 5. 与仓库正文的关系

Discussions 是工作台，不是最终事实的唯一载体。

更稳定的内容应逐步回落到仓库正文：

- 对外公开内容 → `docs/`
- 内部长期规则 → `internal/`
- 正式问题对象 → `issues/`
- 正式演进记录 → `CHANGELOG.md`

Discussions 更适合承载“过程中的结构化记录”，而不是替代仓库正文。
